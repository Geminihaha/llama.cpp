# Adreno 660 OpenCL 최적화 실패 사례 및 분석 기록

이 문서는 `llama.cpp` 최적화 과정에서 발생한 부작용, 성능 저하, 또는 연산 오류 사례를 기록하여 향후 유사한 실수를 방지하기 위한 기술 자료입니다.

---

## 1. [실패] Zero-copy (Map/Unmap) 적용 (2026-04-16)

### 시도 내용
*   `clEnqueueWriteBuffer` 및 `clEnqueueReadBuffer` 호출을 `clEnqueueMapBuffer`와 `memcpy` 조합으로 교체하여 호스트-디바이스 간 데이터 전송 오버헤드를 0으로 만들고자 함.

### 발생한 문제
*   **증상:** 모델 워밍업(Warming up) 단계에서 즉시 **Segmentation Fault** 발생.
*   **원인 분석:**
    1.  **정렬(Alignment) 미준수:** Adreno GPU 드라이버는 Map/Unmap 시 특정 바이트 단위(예: 64바이트)의 정렬을 요구하나, GGML의 텐서 오프셋이 이를 충족하지 못함.
    2.  **동기화 타이밍:** 비동기 큐 작업과 호스트의 직접 메모리 접근 사이의 동기화가 완벽하지 않아 레이스 컨디션(Race Condition) 발생 가능성.

### 교훈 및 대안
*   **교훈:** Adreno의 Unified Memory를 활용할 때는 명시적인 Map/Unmap보다 `CL_MEM_ALLOC_HOST_PTR` 플래그를 통한 드라이버 내부 최적화에 맡기는 것이 훨씬 안전함.
*   **조치:** 해당 최적화 코드는 전량 제거(롤백)함.

## 2. [실패] 공격적인 FP16 및 텍스처 가속 적용 (2026-04-16)

### 시도 내용
*   가장 연산량이 많은 `gemv_noshuffle_q4_k_f32.cl` 커널의 내부 연산을 모두 `half` 타입으로 변경.
*   `read_imagef` (FP32) 대신 `read_imageh` (FP16)를 사용하여 텍스처 엔진 가속.
*   최적화 경로를 타는 텐서 임계값을 512에서 64로 대폭 하향.

### 발생한 문제
*   **증상:** 추론 엔진은 정상 속도로 동작하나, 생성된 문장이 문맥에 맞지 않는 단어들의 나열(Nonsensical output)로 나타남.
*   **원인 분석:**
    1.  **누적 정밀도 오차:** 역양자화(Dequantization) 과정에서 발생하는 `half` 타입의 미세한 오차가 거대한 신경망 레이어를 거치며 증폭됨.
    2.  **텍스처 인덱싱 불일치:** `read_imageh` 사용 시 좌표 계산 방식이 기존 Buffer 방식과 미세하게 달라 가중치 데이터를 잘못 읽었을 가능성.

### 교훈 및 대안
*   **교훈:** LLM 최적화에서 **연산 정확도(Numerical Correctness)**는 성능보다 절대적으로 우선함. 성능을 위해 정밀도를 포기할 때는 반드시 레이어별 검증이 필요함.
*   **조치:** 해당 최적화 사항은 리버트(Revert) 처리함. 향후 핵심 연산부는 FP32를 유지하는 **Mixed Precision** 전략으로 재접근 필요.
